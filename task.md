你是一个资深的全栈桌面应用架构师 + Rust/Tauri 2 工程师 + 浏览器扩展工程师 + 产品设计师。  
请为我从 0 到 1 构建一个“本地作业批改统计应用”的完整项目，目标是：在本地电脑上运行，用于辅助教师在网页批改系统中快速记录学生每道题的对错情况，并自动统计整份作业的数据。

# 一、项目目标

我要一个本地应用，技术栈要求如下：

- 桌面端：Tauri 2
- 后端：Rust
- 前端：React + TypeScript + Vite（如果你认为 Svelte 更合适，也可以说明理由，但默认 React）
- 本地数据库：SQLite
- 浏览器扩展：Chrome Extension Manifest V3
- 扩展与桌面端通信：优先采用最稳定、最容易本地部署的方案（例如 localhost HTTP/WebSocket）。不要先用复杂的 Native Messaging，除非你能证明必要性。
- 应用必须是本地优先（local-first），不依赖云端服务。
- 要求代码结构清晰、模块化、可维护、可扩展。

这个软件主要用来做“作业批改情况统计”。

# 二、核心业务流程

## 1. 学生档案管理

需要有“学生档案”模块，支持：

- 提前一次性录入学生名单
- 支持新增、编辑、删除学生
- 每个学生至少包含：
  - id
  - 姓名（唯一）
  - 学号（可选）
  - 备注（可选）
  - 创建时间
- 后续所有作业项目中的完成人，默认都从这个学生档案中选择或匹配，不需要每次重复输入学生。

## 2. 作业项目管理

左侧是“作业列表”。

每次新建一个作业项目时，我需要输入：

- 作业名称
- 章节信息（可选）
- 题目列表/题号范围

题目结构需要支持：
- 单纯题号：例如 1,2,3,4
- 章节/题号：例如 6.1-1, 6.1-2, 6.2-3
- 后续应支持自定义题目标识，不要把题号写死成纯数字

新建作业后，后台需要初始化这次作业的数据结构：

- 作业
- 题目
- 学生
- 每个学生对每道题的完成状态/批改结果

也就是说，数据库中要能表达：
- 某次作业
- 某个题号
- 某个学生
- 该学生该题是否正确 / 是否未批改 / 是否跳过

## 3. 开始批改

当我点击“开始批改”后，会进入一个批改模式。

此时有一个 Chrome 扩展辅助工作。  
我正在网页批改系统中逐个查看学生作业。网页中当前显示的学生姓名，会出现在页面左上区域（参考我提供的截图，姓名在红色箭头所指位置）。

扩展需要完成：

- 自动检测当前网页中显示的学生姓名
- 尝试把这个姓名发送给桌面应用
- 桌面应用自动匹配到对应学生
- 扩展悬浮显示一个很小的操作面板，置顶，包含：
  - 当前题号显示
  - “对”按钮
  - “错”按钮
  - “上一题”按钮（用于回撤修改上一题状态）
  - 可选：“下一题”按钮（如果有需要）
- 点击“对”或“错”后：
  - 自动记录当前学生、当前题号的结果
  - 自动跳到下一题
- 点击“上一题”后：
  - 撤销上一题记录
  - 返回上一题状态，允许重新标记

## 4. 自动统计逻辑

每录入一次“对/错”，后台都要实时更新：

### 针对题目维度
- 该题已批改人数
- 该题正确人数
- 该题错误人数
- 该题正确率
- 题目错误率

### 针对学生维度
- 当前学生已批改题数
- 当前学生错误题数
- 当前学生当前作业得分

## 5. 自动判分逻辑

如果检测到“本次作业的最后一题也已经录入完毕”，则自动显示该学生本次作业分数。

评分规则如下：

- 全对：100 分
- 错题数占本次题数的 30%（向上取整）以内：95 分
- 超过这个范围：90 分

请把这个规则写成清晰、可配置的评分策略模块，不要写死在 UI 里。

举例：
- 如果本次作业共 10 题，那么 30% = 3，向上取整仍为 3
  - 错 0 题 => 100
  - 错 1~3 题 => 95
  - 错 4 题及以上 => 90
- 如果共 7 题，那么 30% = 2.1，向上取整为 3
  - 错 0 题 => 100
  - 错 1~3 题 => 95
  - 错 4 题及以上 => 90

## 6. 作业统计查看

在桌面应用中，我可以点击某次作业，查看统计结果，包括：

### 总览指标
- 学生平均错题数
- 满分学生数量
- 平均分（可选但建议有）
- 已完成批改人数 / 总人数

### 学生列表视图
- 学生姓名
- 错题数
- 得分
- 每题结果（可点击展开）

### 题目分析视图
- 按题目错误率从高到低排序
- 错误率达到 30% 及以上的题目标红
- 展示每题正确率 / 错误率 / 批改人数

### 可选增强
- 支持搜索学生
- 支持筛选“满分学生”“高错题学生”
- 支持导出 CSV

# 三、Chrome 扩展要求

需要实现一个 Chrome Extension (Manifest V3)。

扩展职责：

1. 在批改网页中识别当前学生姓名
2. 显示悬浮操作条
3. 将“当前学生 + 当前题号 + 对/错操作 + 上一题撤回”等事件发送给桌面端

请优先尝试通过 DOM 文本检测实现学生姓名识别，而不是 OCR。  
因为这是网页系统，不是图片界面。  
如果 DOM 检测失败，可以预留 OCR/截图识别接口，但 MVP 不必先做 OCR。

请设计一个“页面适配层”，因为不同批改系统页面 DOM 结构可能变化。  
要求：
- 将“姓名定位规则”抽象成单独模块
- 支持通过 CSS selector、文本匹配、候选区域扫描等多策略识别姓名
- 支持在设置页中修改 selector（高级设置）

扩展 UI 要求：
- 简洁小巧
- 始终置顶，不遮挡核心批改区域
- 显示当前题号
- 按钮大一些，便于快速点击
- 支持快捷键（例如：
  - A = 对
  - S = 错
  - Z = 上一题
）
- 尽量减少鼠标操作，提高批改效率

# 五、数据库设计要求

请设计合理的 SQLite schema，至少包含：

- students
- assignments
- assignment_questions
- assignment_student_results
- grading_sessions（可选，但建议有）
- app_settings（可选）

其中 assignment_student_results 至少要表达：
- id
- assignment_id
- question_id
- student_id
- status（ungraded / correct / incorrect）
- updated_at

请建立合理索引，保证统计查询性能。

同时请注意：
- 一个学生在同一作业的同一题只能有一条最终状态记录
- 撤销操作需要有可追踪历史，建议设计 action log / event log，或者至少设计可逆操作机制
